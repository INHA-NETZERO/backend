## 팀명: Zero:Wave

## 주제

F&B 식자재 폐기물 제로화를 위한 저전력 sLLM 기반 발주 최적화·탄소 산정 어시스턴트 'Zero-Waste Copilot’

## 선택한 문제

1. 탄소중립에 기여하기 위한 저전력/고효율 LLM 기반 서비스 개발

## 문제 해결을 위한 구체적인 주제 및 선정 이유

### 구체적 주제

**F&B 매장 식자재 폐기물 제로화 - 발주 단계에서 과발주를 사전 차단하는 저전력 sLLM 어시스턴트**

POS·날씨·요일 데이터를 결합해 익일 카테고리별 최적 발주량을 산출하고, 과발주로 예상되던 폐기량을 미리 줄인다. 줄인 폐기량은 카테고리별 전과정 배출계수로 환산되어 '오늘의 절감 탄소(tCO₂eq)'로 점주에게 자연어로 피드백된다.
즉 **해결 시점을 '폐기·처리'가 아니라 '발주'로 끌어올리는 것**이 주제의 핵심이다.

### 선정 이유

**(1) 도메인 -** Scope 3의 관리 사각지대를 가장 효율적인 지점에서 처리

F&B·카페의 신선식품·유제품·베이커리 폐기물은 매장 직접배출(Scope 1·2)이 아니라 식자재 생산·유통 상류 배출(Scope 3 Cat.1)과 폐기 처리 배출(Scope 3 Cat.5)에 걸쳐 있어 관리 사각지대에 놓인다. 특히 폐기의 탄소 영향은 '처리 단계'보다 이미 생산·유통에 투입된 **배태(embodied) 배출**이 훨씬 크다
예) 폐기되는 우유 1L에는 사육·가공·운송 배출이 이미 고정되어 있다. 따라서 폐기 자체를 줄이는 것이 가장 확실한 감축 지렛대이며, 그 폐기를 만드는 원인인 과발주를 막는 것이 가장 최적의 지점이다.

**(2) 규모 -** 작은 감축률도 큰 절대량으로 환산되는 영역

버려지는 음식물은 전 지구 온실가스 배출의 8~10%(IPCC 2019)를 차지하며, 항공 부문 전체 배출의 약 5배에 해당한다. 국내에서도 음식물 쓰레기를 20% 줄이면 온실가스 약 177만 톤이 감소하며, 이는 승용차 약 47만 대의 배출량과 맞먹는다. 참고로 식품 생산 전체는 전 지구 온실가스 배출의 약 1/4~1/3 수준(Poore & Nemecek 2018)으로, 8~10%는 그중 버려지는 몫이다.

**(3) 기술**

정형 수치(매출·재고·날씨) 분석에 초거대 LLM을 쓰면 호출마다 대규모 GPU 전력을 소모해, 탄소를 줄이려다 추론 단계에서 더 많은 탄소를 쓰는 모순이 생긴다. 본 과제는 **수치는 경량 모델, 언어는 소형 모델**로 역할을 분리해 이 모순을 제거하는 것을 기술적 정체성으로 삼는다. LLM을 작게 만드는 데 그치지 않고, 무거운 연산(예측·산정)을 LLM 밖으로 빼내 호출 자체를 줄인다. 부수효과로 LLM의 수치 환각이 원천 차단되어, '탄소 4kg 절감' 같은 정량지표가 공인 배출계수 기반의 검증 가능한 값이 된다.

## 개발을 위해 필요한 것들

### 1. 데이터

- **가상 카페 1개년 POS 매출 데이터(CSV)**
  일자·카테고리·판매량·재고. 실데이터가 없으므로 요일/계절/날씨 상관을 반영한 **합성 데이터 생성**이 사실상 필수
- **기상청 단기예보 공공 API -** 강수·기온을 예측 피처로 사용.
- **카테고리별 배출계수 테이블**
  시연용은 Poore & Nemecek(2018)/Our World in Data의 전과정(LCA) 평균값을 정적 룩업테이블에 적재
  검증된 대표값 예) 소고기 ≈60, 치즈 20+, 닭/돼지 ≈6/7 kgCO₂eq/kg, 우유 ≈3 kgCO₂eq/kg

### 2. 백엔드 / 데이터 파이프라인

- Java(Spring Boot 3.x)
  외부(날씨)·내부(POS) 데이터 수집·정제·피처 생성 ETL → **EC2** 1대에서 실행
- 정제 데이터·산정 로그 저장 → **DynamoDB**

### 3. 결정적 분석 계층 (비-LLM, 정량지표의 신뢰성을 좌우)

- **경량 수요예측 모델**
  **전 매장 통합 글로벌 모델 1개**(매장 ID·카테고리를 피처로 한 LightGBM)
  요일·공휴일·강수·기온을 피처로 카테고리별 일 수요 예측
  정확도는 WAPE(또는 1−MAPE)로 관리하되, 신선식품은 변동이 커서 카테고리별 목표 차등화 권장
- **발주 최적화 로직**
  유통기한 짧은 단일기간 품목에는 표준 안전재고 공식보다 **신문팔이(newsvendor) 모형**이 적합
  과소(결품) 비용 vs 과잉(폐기) 비용의 임계분위로 최적 발주량을 구해, 폐기비용이 큰 품목일수록 발주를 자동 보수화한다.
- **탄소 산정 모듈**
  `ΔE = Σ ΔQᵢ × (EF_prod,ᵢ + EF_waste,ᵢ)` 산정식과 룩업테이블 조회($O(1)$)
  최소 보장 감축(처리 회피분)과 잠재 감축(생산 회피분 포함) 라벨을 분리
- 실행 형태
  이 계층(날씨 수집 → 예측 → 발주 → 탄소 산정)은 하루 한 번 도는 배치이므로 **Lambda + EventBridge**로 스케줄 실행 (scale-to-zero = 상시 가동 없는 저전력 구조)

### 4. 언어 계층 (sLLM, 설명·대화 전용)

- **Python 3.10+, FastAPI** - 추론 서빙
- **Hugging Face Transformers + 양자화(BitsAndBytes 4·8-bit)**
  한국어 가능한 소형 오픈소스 모델을 self-host(수치 연산은 안 시키고 2~3문장 설명만 담당하므로 소형으로 충분) → **EC2 + Inferentia(Inf2)**에서 서빙해 추론 전력 최소화
- **RAG (LangChain 등)**
  ②의 결과(요일·날씨·발주이력)를 근거로 검색해 답변
- **시맨틱 캐싱**
  유사 질의를 임베딩 유사도로 매칭해 재연산 방지. 적중률을 효율 지표로 측정.

### 5. 프론트엔드

- **React 또는 Vue.js 기반 경량 대시보드 + 챗봇 UI** - 발주 가이드, 절감 리포트, 자연어 질의 화면 구현
  → **AWS Amplify**로 git 연동 자동 배포·호스팅

### 6. 하드웨어 / 측정 도구

- GPU 의존성 없음 - 프로토타입은 CPU만으로 구동 가능하며, 추론 최적화를 위해 **Inferentia**를 사용
- **저전력 입증용 계측**
  동일 시나리오를 '전부 LLM' 방식과 본 설계로 처리할 때의 LLM 호출 횟수·총 토큰·응답 지연(ms)·캐시 적중률을 **CloudWatch**로 수집·대시보드화
